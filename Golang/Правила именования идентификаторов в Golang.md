

## Именование идентификаторов
### Жёсткие ограничения (код не скомпилируется при нарушении):
- Разрешены только **буквы Unicode, цифры и подчёркивания**.
- Идентификатор **не может начинаться с цифры**.
- **Запрещено использовать ключевые слова Go**: `break`, `func`, `type`, `interface`, `chan`, `defer`, `go`, `map`, `range`, `switch` и др.

### Рекомендованные правила именования идентификаторов 

| Правило                                                         | Пример ✅                           | Пример ❌                           |
| --------------------------------------------------------------- | ---------------------------------- | ---------------------------------- |
| **camelCase** для внутренних, **PascalCase** для экспортируемых | `apiKey`, `LoadUser`               | `api_key`, `load_user`             |
| Акронимы в едином регистре                                      | `HTTPClient`, `parseXML`, `userID` | `HttpClient`, `parseXml`, `userId` |
| Только ASCII-символы                                            | `pi`, `naiveBayes`                 | `π`, `naïveBayes`                  |
| Не конфликтовать с builtin-типами и функциями                   | `clearQueue`, `data`               | `clear`, `int`                     |
| Не включать тип в имя                                           | `count`, `results`                 | `countInt`, `resultsSlice`         |
| Избегать имён стандартных пакетов                               | `payload`, `logger`                | `json`, `url`, `mail`              |


### Экспортируемые и внутренние идентификаторы

- Идентификатор с **заглавной буквы** — экспортируется (виден вне пакета).
- С **прописной буквы** — доступен только внутри пакета.
- По умолчанию делайте идентификаторы **неэкспортируемыми**.
- Экспортируйте только то, что действительно нужно использовать снаружи.
- В пакете `main` почти все имена должны быть внутренними (исключение — поля структур для `encoding/json` и подобных пакетов, использующих рефлексию).


### Длина и описательность имён

**Принцип:** чем дальше используется идентификатор от места объявления — тем описательнее должно быть имя.
```go
// Короткое имя допустимо в узкой области видимости
for _, p := range people {
    sum += p.Age  // p — понятно в контексте range
}

// Более описательные имена для переменных с большей областью
var count, sum int  // лучше, чем c и s
```




## Именование пакетов

### Конвенции:

- Только **строчные латинские буквы и цифры**, без подчёркиваний.
- не начинать имена пакетов с символов `_`, `.`
- Короткие, понятные, отражающие суть: `orders`, `slug`, `auth`.
- Много слов — слитно: `ordermanager` ✅, `orderManager` ❌.
- Допустимы устоявшиеся сокращения: `strconv`, `expvar`.


|Проблема|Пример ❌|Решение ✅|
|---|---|---|
|Начинается с цифры|`package 2fa`|`package twofa`|
|Нестандартный регистр|`OrderManager`|`ordermanager`|
|Слишком абстрактно|`utils`, `helpers`|`validation`, `formatting`|
|Конфликт со stdlib|`package url`|`package links`|
|Специальные имена|`internal`, `vendor`, `testdata`|`internalauth`, `supplier`|
|Игнорируется тулингом|`_cache`, `.hidden`|`cache`, `hidden`|


## Именование файлов
**Идеал:** одно слово, нижний регистр, отражает содержание: `server.go`, `cookie.go`, `status.go`.

**Если нужно несколько слов:** выберите один стиль и придерживайтесь его:

- Слитно: `routingindex.go` _(рекомендуется)_
- Через подчёркивание: `routing_index.go`

### Специальные суффиксы (не используйте без необходимости):
|Суффикс|Эффект|
|---|---|
|`_test.go`|Запускается только через `go test`|
|`_linux.go`, `_windows.go`|Компилируется только для указанной ОС|
|`_amd64.go`, `_arm64.go`|Компилируется только для указанной архитектуры|
|`_` или `.` в начале|Игнорируется тулингом Go|



## Именование функций
### Избегайте «болтовни» (redundancy)
Не дублируйте имя пакета в именах функций, типов и методов.
```go
// ❌ Избыточно
customer.NewCustomer()
customer.CustomerOrders()
customer.CustomerAddress

// ✅ Лаконично и читаемо
customer.New()
customer.Orders()
customer.Address
```


### Ресиверы методов
- Имя получателя должно быть **коротким** (1–3 символа), обычно аббревиатура типа.
- **Будьте последовательны**: все методы одного типа должны использовать одно и то же имя получателя.
- Не используйте `this`, `self`, `me` — это не идиоматично для Go.
```go
type Order struct { Items int }

// ✅ Хорошо
func (o *Order) Validate() bool { return o.Items > 0 }

// ❌ Избегайте
func (order *Order) Validate() bool { ... }  // слишком длинно
func (self *Order) Validate() bool { ... }   // generic-имя
```


### Геттеры и сеттеры
Go обычно **не нужны** геттеры/сеттеры — обращайтесь к полям напрямую.

**Исключение:** поле неэкспортируемое, но нужен доступ извне.
```go
type Customer struct {
    address string  // неэкспортируемое
}

// Геттер — без префикса Get
func (c *Customer) Address() string {
    return c.address
}

// Сеттер — с префиксом Set
func (c *Customer) SetAddress(addr string) {
    c.address = addr
}
```


### Использование префиксов
Префиксы в названии функций помогают понять намерение автора кода, не заглядывая внутрь функции. Вот самые популярные:

#### 1. Предикаты (Логические вопросы)

Возвращают логическое значение (`true`/`false`).
- **`is`**: Состояние объекта (`isAlive`, `isAdmin`).
- **`has`**: Наличие свойства или элемента (`hasLicense`, `hasAccess`).
- **`can`**: Возможность совершить действие (`canEdit`, `canPost`).
- **`should`**: Рекомендация или условие (`shouldRefresh`).

#### 2. Жизненный цикл и события
- **`on`**: Обработчик события (`onClick`, `onLoad`).
- **`handle`**: Логика обработки данных или ошибок (`handleRequest`, `handleError`).
- **`before` / `after`**: Хуки, выполняющиеся до или после основного действия.

#### 3. Получение и установка (Accessors)
- **`set`**: Устанавливает значение.
- **`fetch` / `request`**: Подразумевает асинхронный запрос к API или базе данных (длительная операция).
- **`find`**: Поиск элемента (может вернуть `null`, если ничего не найдено).

#### 4. Трансформация и создание
- **`to`**: Конвертация из одного типа в другой (`toString`, `toJson`).
- **`as`**: Представление объекта в другом виде (`asArray`).
- **`new` / `create`**: Создание нового экземпляра объекта.
- **`parse`**: Разбор строки или данных в структуру.
    
#### 5. Требования к выполнению
 `Must` - Функция обязана выполниться успешно, иначе - паника 










##  Именование интерфейсов
- Интерфейс с **одним методом**: имя = метод + суффикс `-er` (или аналог). 
   Например: `io.Reader`, `io.Writer`, `fmt.Stringer`.
```go
type Speaker interface {
    Speak() string
}

type Authorizer interface {
    Authorize(ctx context.Context, action string) error
}
```

- Не добавляйте слово `Interface` в имя: `UserInterface` ❌ → `User` ✅ (если это интерфейс).
