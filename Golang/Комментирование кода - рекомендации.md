
## Пакеты

Комментарий уровня пакета пишется в самом начале исходного go-файла  - перед директивой  **package** . Он  содержит информацию, относящуюся к пакету в целом, и в целом определяет ожидания от пакета. Полезно, чтобы комментарий давал краткий обзор наиболее важных частей API, при необходимости ссылаясь на другие комментарии в документации.
Для пакетов, содержащих несколько файлов, комментарий к пакету должен находиться только в одном исходном файле. Если комментарии к пакету содержатся в нескольких файлах, они объединяются в один большой комментарий для всего пакета.



## Команды

Комментарий к команде в пакете аналогичен, но описывает поведение программы, а не символы Go в пакете. Первое предложение обычно начинается с названия самой программы, написанного с заглавной буквы, поскольку оно находится в начале предложения.

## Типы

В комментарии к типу следует объяснять, что представляет собой каждый экземпляр этого типа или что он предоставляет. Если API простой, документация может быть довольно короткой.
По умолчанию ожидается, что тип безопасен для использования только одной горутиной одновременно. Если тип предоставляет более надежные гарантии, это должно быть указано в комментарии к документации. Например:
```go
package regexp

// Regexp is the representation of a compiled regular expression.
// A Regexp is safe for concurrent use by multiple goroutines,
// except for configuration methods, such as Longest.
type Regexp struct {
    ...
}
```

Также следует комментировать  значение zero-value для типа.
```go
package bytes

// A Buffer is a variable-sized buffer of bytes with Read and Write methods.
// The zero value for Buffer is an empty buffer ready to use.
type Buffer struct {
    ...
}
```


Для структуры с экспортируемыми полями либо в комментарии к типу, либо в комментариях к каждому полю должно быть пояснено значение каждого экспортируемого поля.
```go
package io

// A LimitedReader reads from R but limits the amount of
// data returned to just N bytes. Each call to Read
// updates N to reflect the new amount remaining.
// Read returns EOF when N <= 0.
type LimitedReader struct {
    R   Reader // underlying reader
    N   int64  // max bytes remaining
}
```



## Функции

Комментарии к функции запуоняются неповредственно перед объсявлением самой функции. 
В в таких комментах должно быть пояснено, что функция возвращает или, для функций, или что она делает (для функций с побочкой). Именованные параметры и результаты могут быть указаны непосредственно в комментарии, без использования специального синтаксиса, такого как обратные кавычки.


В комментариях к документации не следует описывать внутренние детали, такие как алгоритм, используемый в текущей реализации. Лучше оставить это для комментариев внутри тела функции. Указание асимптотических временных или пространственных ограничений может быть уместным, если эта деталь особенно важна для вызывающих функций.


## Константы

Синтаксис объявлений в Go позволяет группировать объявления, в этом случае один документационный комментарий может вводить группу связанных констант, при этом отдельные константы документируются только короткими комментариями в конце строки. Например:
```go
package scanner // import "text/scanner"

// The result of Scan is one of these tokens or a Unicode character.
const (
    EOF = -(iota + 1)
    Ident
    Int
    Float
    Char
    ...
)
```

Иногда группе вообще не нужны комментарии к документу. Например:
```go
package unicode // import "unicode"

const (
    MaxRune         = '\U0010FFFF' // maximum valid Unicode code point.
    ReplacementChar = '\uFFFD'     // represents invalid code points.
    MaxASCII        = '\u007F'     // maximum ASCII value.
    MaxLatin1       = '\u00FF'     // maximum Latin-1 value.
)
```

Типизированные константы отображаются рядом с объявлением их типа, и поэтому часто опускают комментарий к группе констант, предпочитая комментарий к типу. Например:
```go
package syntax

// An Op is a single regular expression operator.
type Op uint8

const (
    OpNoMatch        Op = 1 + iota // matches no strings
    OpEmptyMatch                   // matches empty string
    OpLiteral                      // matches Runes sequence
    OpCharClass                    // matches Runes interpreted as range pair list
    OpAnyCharNotNL                 // matches any character except newline
    ...
)
```


## Переменные

Правила работы с переменными такие же, как и с константами. Например, вот набор сгруппированных переменных:
```go
package fs

// Generic file system errors.
// Errors returned by file systems can be tested against these errors
// using errors.Is.
var (
    ErrInvalid    = errInvalid()    // "invalid argument"
    ErrPermission = errPermission() // "permission denied"
    ErrExist      = errExist()      // "file already exists"
    ErrNotExist   = errNotExist()   // "file does not exist"
    ErrClosed     = errClosed()     // "file already closed"
)
```




## Правила ведения коментариев 
-  **Используйте стандартные форматы**
 Go имеет собственный инструмент `go doc`, который генерирует документацию на основе комментариев. Чтобы он работал правильно, следуйте этому формату:

- **Комментируйте, что делает код, а не как**
Хороший код сам по себе должен быть достаточно читаемым, чтобы было понятно, как он работает. Комментарии должны объяснять **почему** было принято то или иное решение, или **что** делает этот блок кода, если это не очевидно.
Плохо:
```go
// Increment the counter by one.
counter++
```
Это и так очевидно.

Хорошо:
```go
// We use a mutex here to prevent race conditions during concurrent access.
mu.Lock()
defer mu.Unlock()
counter++
```

Это объясняет причину использования мьютекса.

- **Комментируйте экспортируемые (Public) элементы**
Все экспортируемые функции, переменные, типы и константы (начинающиеся с заглавной буквы) должны иметь комментарии. Это важно для создания хорошей документации, которая помогает другим разработчикам использовать ваш код.
```go
// User represents a user in the system with their name and ID.
type User struct {
    ID   int
    Name string
}

// GetUserByID retrieves a user from the database by their unique ID.
func GetUserByID(id int) (*User, error) {
    // ...
}
```

- **Избегайте лишних комментариев**
Слишком много комментариев может быть хуже, чем их полное отсутствие. Не комментируйте очевидные вещи. Если код сложен, лучше его рефакторить, чтобы он стал понятнее, а уже затем добавлять пояснения.

Не хорошо и не красиво - писать лишнее:
```go
// a is the first value, b is the second.
func add(a, b int) int {
    // Return the sum of a and b.
    return a + b
}
```

- **Используем `//` для однострочных комментариев и `/* ... */` для многострочных**

- `//` используется для большинства комментариев, включая те, что описывают функции и переменные.
- `/* ... */` полезны для временного отключения блока кода или для обширных пояснений. Однако они используются гораздо реже, чем `//`.

- **Документируем ошибки**
Если функция может возвращать ошибку, поясните, какие именно ошибки она может вернуть и при каких условиях. Это особенно полезно для тех, кто будет использовать вашу библиотеку.

```go
// OpenFile opens the file at the given path.
// It returns an error if the file does not exist or if permissions are insufficient.
func OpenFile(path string) (*os.File, error) {
    // ...
}
```
