### Общие Концепции и Принципы
- **Масштабируемость (Scalability)** - Способность системы справляться с **увеличением нагрузки** (пользователей, данных, транзакций) за счет добавления ресурсов (например, серверов). Различают **вертикальное** (увеличение мощности одного сервера) и **горизонтальное** (добавление большего числа серверов) масштабирование.
- **Надежность (Reliability)** - Вероятность того, что система будет **работать без сбоев** в течение определенного периода времени при заданных условиях. Достигается за счет **избыточности** (redundancy) и механизмов восстановления после ошибок.
- **Доступность (Availability)** - Доля времени, в течение которой система **доступна и работоспособна** для пользователей. Часто измеряется в «девятках» (например, $99.999\%$ — "пять девяток").
- **Отказоустойчивость (Fault Tolerance)** - Способность системы **продолжать функционировать** даже при возникновении **сбоев** в одном или нескольких ее компонентах. Достигается за счет дублирования и автоматического переключения.
- **Производительность (Performance)** - Насколько быстро система может выполнять свою работу. Измеряется показателями, такими как **задержка (latency)** — время от запроса до ответа, и **пропускная способность (throughput)** — количество операций, выполняемых за единицу времени.
- **CAP-Теорема** - Фундаментальный принцип, утверждающий, что в распределенной системе невозможно одновременно гарантировать все три свойства: **Согласованность (Consistency)**, **Доступность (Availability)** и **Устойчивость к разделению (Partition Tolerance)**. Можно выбрать только два из трех.
- **Задержка (Latency)** - Время, необходимое для выполнения операции, например, время, которое требуется пакету данных для перехода от источника к получателю и получения ответа.
- **Пропускная Способность (Throughput)** - Количество единиц работы (запросов, транзакций), которое система может обработать за единицу времени.
- **RPS (Requests Per Second)** — это показатель, который измеряет **количество запросов, которые система, сервис или приложение способно обработать за одну секунду**.


### Архитектурные Паттерны и Стили
- **Монолитная Архитектура (Monolithic Architecture)** - Традиционный подход, при котором **все компоненты** приложения (пользовательский интерфейс, бизнес-логика, доступ к данным) объединены в **единый** выполняемый блок.
- **Микросервисная Архитектура (Microservices)** - Архитектурный стиль, при котором большое приложение разбивается на набор **небольших, независимых сервисов**, каждый из которых выполняет определенную бизнес-функцию, может быть разработан, развернут и масштабирован **отдельно**.
- **Серверless (Serverless)** - Модель облачных вычислений, при которой провайдер **динамически управляет выделением ресурсов**, и оплата взимается только за фактическое использование (например, AWS Lambda, Azure Functions). Разработчик не управляет серверами.
- **SOA (Service-Oriented Architecture)** - Архитектура, в которой приложение состоит из **слабосвязанных, автономных сервисов**, взаимодействующих между собой, часто через общую шину (Enterprise Service Bus). Предшественник микросервисов.
- **Балансировщик Нагрузки (Load Balancer)** - Устройство или программное обеспечение, которое **распределяет входящие сетевые запросы** между группой (пулом) серверов. Улучшает доступность и масштабируемость.
- **Шлюз API (API Gateway)** - Единая точка входа для всех клиентов, которая управляет маршрутизацией запросов к соответствующим микросервисам, аутентификацией, ограничением скорости и другими сквозными функциями.
- **Шина Сообщений / Брокер Сообщений (Message Queue / Broker)** - Компонент, который обеспечивает **асинхронное взаимодействие** между сервисами, сохраняя сообщения до тех пор, пока их не обработает получатель (например, RabbitMQ, Kafka, AWS SQS).


### Компоненты Хранения Данных
- **Репликация (Replication)** - Процесс **копирования данных** с одного сервера (Мастер) на другие (Реплики) для повышения надежности, доступности и производительности (чтение может выполняться с реплик).
- **Секционирование / Шардирование (Sharding)** - Метод **горизонтального разделения данных** в базе данных, при котором данные разбиваются на более мелкие, независимые части (шарды), хранящиеся на разных серверах, для масштабирования.
- **Кэширование (Caching)** - Хранение часто запрашиваемых данных во **временном, более быстром хранилище** (кэше), чтобы уменьшить задержку при доступе и снизить нагрузку на основную базу данных.
- **CDN (Content Delivery Network)** - Географически распределенная сеть прокси-серверов, которая **кэширует статический контент** (изображения, видео, CSS, JS) ближе к конечным пользователям для уменьшения задержки.
- **SQL/NoSQL** - **SQL (Реляционные БД)**: Используют структурированную схему, язык SQL и поддерживают транзакции ACID. **NoSQL (Нереляционные БД)**: Гибкие схемы, подходят для горизонтального масштабирования и больших объемов данных (например, MongoDB, Cassandra).
- **ACID** - Набор свойств, гарантирующих, что транзакции базы данных обрабатываются **надежно**: **А**томарность, **С**огласованность, **И**золированность, **Д**олговечность. Обычно связаны с SQL-базами.
- **BASE** - Набор принципов, характерных для NoSQL-баз, которые жертвуют строгой согласованностью ради доступности и устойчивости к разделению: **B**asically **A**vailable, **S**oft state, **E**ventual consistency.



### Взаимодействие и Безопасность
- **API (Application Programming Interface)** - Набор правил и протоколов, который позволяет различным программным компонентам **взаимодействовать друг с другом**. Определяет методы и форматы данных.
- **REST (Representational State Transfer)** - Популярный архитектурный стиль для построения **распределенных систем**, использующий стандартные HTTP-методы (GET, POST, PUT, DELETE) для обмена данными, часто в формате JSON или XML.
- **RPC (Remote Procedure Call)** - Протокол, позволяющий программе **выполнять функцию или процедуру** в другом адресном пространстве (обычно на другом сервере), как если бы она была локальной.
- **OAuth 2.0** - Стандарт **авторизации**, который позволяет стороннему приложению получать ограниченный доступ к защищенным ресурсам пользователя без необходимости раскрывать его учетные данные.
- **JWT (JSON Web Token)** - Компактный, URL-безопасный механизм для **передачи информации между сторонами** в виде объекта JSON. Часто используется для аутентификации и авторизации.
- **Идемпотентность (Idempotence)** - Свойство операции, при котором ее **многократное выполнение** дает **тот же результат**, что и однократное (например, HTTP-запросы GET, PUT и DELETE по своей природе идемпотентны).



### Требования и Проектирование

- **FR (Functional Requirements)** - **Функциональные Требования** — описывают, **что система должна делать**. Это конкретные функции и поведение, которые конечный пользователь ожидает от системы (например, "Система должна позволять пользователю авторизоваться с помощью email и пароля").
- **NFR (Non-Functional Requirements)** - **Нефункциональные Требования** — описывают, **как система должна работать**. Они касаются **качества** системы (например, Производительность, Надежность, Безопасность, Масштабируемость, Удобство использования). В системном дизайне часто определяют ключевые **ограничения**.
- **Use Case (Сценарий Использования)** - Описание **взаимодействия** пользователя (Актора) с системой для достижения определенной цели. Используется для определения функциональных требований.
- **Design Document (Проектный Документ)** - Подробный технический документ, описывающий **архитектурные решения**, компоненты, интерфейсы, хранение данных и нефункциональные требования для реализации системы.
- **TCO (Total Cost of Ownership)** - **Общая Стоимость Владения** — включает все расходы, связанные с приобретением, развертыванием, использованием, обслуживанием и поддержкой системы или компонента в течение всего срока его службы.
- **POC (Proof of Concept)** - **Подтверждение Концепции** — небольшой эксперимент, направленный на проверку **жизнеспособности** конкретной идеи, технологии или архитектурного решения.

### Метрики и Операции (Ops)
- **SLA (Service Level Agreement)** - **Соглашение об Уровне Обслуживания** — официальный договор между поставщиком и клиентом, определяющий **минимальный гарантированный уровень доступности** и качества сервиса (например, $99.9\%$ доступности).
- **SLO (Service Level Objective)** - **Цель Уровня Обслуживания** — конкретная, измеримая цель, которую должен достичь сервис, чтобы выполнить SLA (например, "Задержка $95\%$ запросов должна быть менее 200 мс").
- **SLI (Service Level Indicator)** - **Индикатор Уровня Обслуживания** — конкретная метрика, используемая для измерения соответствия SLO (например, процент успешных запросов, средняя задержка).
- **MTBF (Mean Time Between Failures)** - **Среднее Время Между Отказами** — метрика, показывающая, как долго система или компонент **работает без сбоев** (показатель надежности).
- **MTTR (Mean Time To Recovery)** - **Среднее Время Восстановления** — метрика, показывающая, сколько времени требуется для **восстановления системы после сбоя** (показатель ремонтопригодности).
- **IdP (Identity Provider)** - **Поставщик Идентификации** — система, которая создает, поддерживает и управляет информацией об идентификации и аутентификации пользователя (например, Google, Okta, Azure AD).


### Безопасность и Тестирование
- **DoS/DDoS** - **Отказ в Обслуживании (Denial of Service)** / **Распределенный Отказ в Обслуживании (Distributed DoS)** — атаки, направленные на **перегрузку системы** запросами с целью сделать ее недоступной для легитимных пользователей.
- **DRP (Disaster Recovery Plan)** - **План Аварийного Восстановления** — набор политик и процедур, описывающих, как компания будет **восстанавливать доступ** к своим ИТ-инфраструктурам и данным после крупного сбоя (стихийное бедствие, кибератака).
- **RPO (Recovery Point Objective)** - **Целевая Точка Восстановления** — максимальное количество данных, которое система может позволить себе **потерять** в случае сбоя. Определяет частоту резервного копирования.
- **RTO (Recovery Time Objective)** - **Целевое Время Восстановления** — максимальное **время простоя**, допустимое после сбоя. Определяет, как быстро система должна быть восстановлена.
- **Penetration Testing (Пентест)** - **Тестирование на Проникновение** — симуляция кибератаки на систему для оценки ее **безопасности** и выявления уязвимостей.
